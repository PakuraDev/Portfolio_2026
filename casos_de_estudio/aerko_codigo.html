Arquitectura SPA, Enrutamiento y Reactividad Nativa

A ver, empezamos mal porque explicar esto en cristiano va a requerir una creatividad demasiado grande. Ofu, es que tengo que explicar lo que es un framework, un spa, un router, un bus de eventos, un webcomponent, shadow dom, el propio dom, y muchas cosas más... Voy a hacer una cosa, voy a explicar rápidamente, el que sepa de lo que hablo se la puede saltar directamente.

Explicación rápida

Bien, para explicar todo vamos a imaginar que construir una app es como gestionar un teatro. La mayoria de las apps se hacen usando un framework (como React, Angular (si alguien usa eso) o Astro si sois gigachads), que básicamente es como ir al IKEA y comprarte un mueble con la madera ya cortada, los tornillos y un manual de instrucciones. Es súper cómodo y rápido. Pero yo soy un obseso del rendimiento (o un masoquista, ambas definiciones son correctas), así que yo use JS Vanilla, que en este contexto es exactamente lo mismo que coger el coche, ir al bosque, fabricar un hacha, talar un árbol, lijar la madera, y construir los muebles con mis propias manos.

Si habéis entendido esto sin necesidad de ver un tutorial o preguntarle a ChatGPT, entonces vais muy bien. Pasamos con el DOM, que básicamente sería el escenario del teatro con sus decorados. En una web tradicional, cada vez que haces clic en un botón, el teatro cierra el telón y cambian el escenario. Pero mi web es una SPA (Single Page Application), lo que significa que el telón nunca se cierra. Si cambias de pestaña, Flash (o cualquier cosa con una velocidad inhumana que os podaís imaginar) cambia el decorado a la velocidad de la luz mientras tu sigues mirando sin necesidad de recargar.

Ahora vamos con el router, que más o menos es como un portero de discoteca. Para gestionar los cambios de decorado sin que nadie se cuele, programe mi propio router. Cada vez que intentas ir a una sección privada de la app, el router te intercepta y te dice "¿Tienes la sesión criptográfica abierta?" o en equivalente en andaluz "Quillo, ¿Eres del betis o del sevilla?". Si has metido el pin (o eres del betis) te deja pasar. Si no, te da una patada y te manda directo a las oficinas de Hacienda para que te hagan una revisión sorpresa (Por cierto, aclaro rápidamente que no me refiero al router de vuestras casas, si no un pequeño script).

Vamos con el bus de eventos (o EventBus), al que voy a llamar "megáfono". Como yo no vo al IKEA, las diferentes partes de la app (muebles) no vienen contectadas entre si, pero claro, yo necesito que la sección de nutrición sepa que Juan ha apuntado su peso para volver a calcular sus calorias. Para eso programé en bus de eventos, que básicamente es como un megáfono que le grita lo que haces al resto de la app.

Ahora vamos con lo que posiblemente sea más difícil de explicar, los webcomponets y el Shadow DOM. Básicamente es como si cada componente de la app fuera un tupper. Vamos a pasar del teatro a la cocina de un bar. A veces el diseño (css) es un caos. A veces tienes un botón de rojo en la página de inicio, pero por un error de cascada se pinta de rojo en los ajustes, cuando los botones de los ajustes son azules. Explicado con el bar, es como si el camarero apunta 3 pedidos, pero el chef se confunde y le pone a todos los platos cebolla, cuando solo uno de ellos la llevaba. Para evitar eso, el camarero le da los 3 tickets por separados, y el chef mete cada comida en un tupper hermético para que no se mezclen los ingredientes. Cada tupper es un webcomponent, y el shadow DOM es la comida en el interior.

Y con estos conceptos básicos, ahora viene el examen. Voy a empezar a hablar en friki, así que si no entendéis lo siguiente, no os preocupéis, es normal.

El router guardián

Hacer un router personalizado basado en window.location.hash es fácil. Lo difícil es que ese router sea un paranoico de la seguridad. En Aerko_, el router, antes de renderizar un solo elemento intercepta la ruta, le pregunta al servicio de autenticación si la bóveda criptográfica está abierta (authService.hasPin()) para mandar a la ruta privada o al login. Además, maneja el ciclo de vida del usuario, es decir, que si eres nuevos te secues... lleva amablamente al onboarding. Todo esto inyectando HTML crudo y/o clases con replaceChildren() para que la pantalla no parpadee siquiera.

Persistencia de datos y criptografía (Privacy by Design)

Aquí la cosa se pone seria. Tu móvil (u el dispositivo donde uses la app, a lo mejor alguien la usa en un ipad o incluso en un mini pc, que por poder pueden) es la base de datos, lo que significa que si alguien te lo roba o lo enchufa a un pc (sabiendo tu pin), tus datos pueden ser vulnerables. Aunque es opcional (dado que no son datos sensibles), monté una arquitectura de bóvedas encriptadas de extremo a extremo usando IndexedDB y la Web Crypto API. Si la activas, tus datos estarán más fuertes que el vinagre, mientras que si no, son vulnerables (aunque para ser sinceros, no creo que a nadie le importe que se filtren sus calorias diarias).

Bóvedas y el Wrapper Transparente

No voy a mentir, el que hizo IndexedDB de seguro odiaba su trabajo. Literalmente es insufrible por su sistema de callbacks, así que directamente hice una clase wrapper con promesas para hacerlo transaccional (también plantee la posibilidad de usar sqlite en wasm, pero como he tocado más veces IndexedDB fui a lo seguro). Obviamente dividí la base de datos en diferentes bóvedas, como la "Public_store" (para cosas tontas como el idioma o el tema) y bóvedas acorazadas como la "user_vault", "training_vault", etc.

La gracia viene con la DX (Developer Experience o Experiencia de desarrollador). El resto del código de la app no sabe de encriptación, solo le dice a la base de datos "Quillo, ¿Me guardas esto?", siendo la clase de la base de datos la que intercepta "eso" (el objeto), comprueba si va a una bóveda protegida, y si es así, genera un vector de inicialización (IV) único de 12 bytes que se crifra en AES-GCM al vuelo. Aunque ha sido muy friki, el resultado básicamente es pasar de "He comido patatas" a "iv: new Uint8Array([122, 201, 8, 45, 19, 110, 2, un largo etc que no pienso escribir])". Para que veáis a qué nivel de intercepción me refiero, os dejo un snippet sacado del código:

JAVASCRIPT
// src/core/db/index.js (Extracto de intercepción)
async put(storeName, data) {
    // 1. Bypass si la bóveda es pública o no hay clave en RAM
    if (!this._isProtected(storeName) || !cryptoService.key) {
        return this._transaction(storeName, 'readwrite', store => store.put(data));
    }

    // 2. Intercepción y cifrado al vuelo (AES-GCM)
    const { id, ...payload } = data;
    const encryptedData = await cryptoService.encrypt(payload);
    
    const recordToSave = { 
        id: id, 
        data: encryptedData // Solo se guarda el IV y el buffer cifrado
    };
    
    // 3. Inserción segura
    return this._transaction(storeName, 'readwrite', store => store.put(recordToSave));
}

Memoria RAM y Paranoia

Ahora, ¿De dónde sale la clave para cifrar eso? Pues del pin que pones al entrar. Usando PBKDF2 (que si bien me hubiera gustado usar argon2, ya lo veía totalmente excesivo, además de que nuevamente es un wasm y hubiese complicado todo, porque argon2 es hash), 100,000 iteraciones y un "Salt" aleatorio, se deriva una clave AES-GCM que haría llorar a cualquier hacker ruso. Y aquí viene la genialidad (más que genialidad es seguridad básica xd), esa clave solo vive en la memoria RAM. Cada vez que bloqueas o te sales de la app, se hace this.key=null y la clave desaparece. Sin tu pin es matemáticamente imposible. Tanto es así que el sistema está cerca de ser quantum-proof.

Por último, para saber si el pin es correcto sin guardar el pin real (lo cual sería sintoma de idiotez severa), se usa el patrón "Canary". Guardo un objeto cifrado que dice "{ secret: 'AERKO_SECURE' }", y cuando el usuario mete su pin, se intenta descifrar al canario. Si me devuelve 'AERKO_SECURE', sé que eres tú y te dejo pasar. Si no, el canario muere y tú te quedas fuera. Esto por otra parte no es a prueba de balas, obviamente cualquier persona lo suficientemente aburrida podría intentar fuerza bruta, pero no creo que a nadie le compense robar un móvil, su pin, extraer los datos de la app (que encima están en el navegador y eso jode el triple para exportar) y descifrarlos via fuerza bruta, y todo para saber tu rutina de pierna (o si efectivamente te la saltas, que te he pillado).

Web Workers y capacidad Offline

Siguiendo mi filosofía de libertad individual, y para permitir que cualquier dispositivo pueda usar la función, no podía permitir meter un solo modelo y obligar a la gente a usarlo, por lo que implementé un sistema de descarga donde puedas elegir entre el Lite, Full y Heavy (junto con un sistema de recomendación) y guardarlo en la caché del navegador. Pero hay un pequeño problema: Si meto a una IA (que ya de por si es pesada) a analizar un video en el mismo hilo en el que se procesa la pantalla, la app se congela. ¿La solución? Meter la IA en un Web Worker para que trabaje en segundo plano.

Pero ojo, porque si creíais que aquí se acababa mi miseria es que no habéis usado una librería hecha por google. Mediapipe asume que está en una página normal (cosa que tampoco puedo echar en cara, la verdad) y busca objetos como "window" o "document", pero claro, en el web worker esos objetos no existen, así que no me quedó otra, crear un DOM falso para que la IA se crea que está en una página normal. Voy a dejar por aquí un pequeño extracto para que veáis el nivel de locura:

JAVASCRIPT
// src/modules/training_lab/views/pose-worker.js (Extracto de interceptación)

// TRUCO: El 'document' falso (Nivel experto con interceptor de scripts)
if (typeof document === 'undefined') {
    self.document = {
        createElement: function(tag) {
            // Si la IA intenta crear un canvas, le damos un OffscreenCanvas
            if (tag === 'canvas') {
                const canvas = new OffscreenCanvas(1, 1);
                canvas.setAttribute = function() {};
                if (!canvas.addEventListener) canvas.addEventListener = function() {};
                return canvas;
            }
            
            // Creamos un objeto falso para cualquier otro elemento (ej. scripts)
            return {
                tagName: tag.toLowerCase(),
                src: '',
                listeners: {},
                setAttribute: function(name, value) { if (name === 'src') this.src = value; },
                addEventListener: function(type, listener) { this.listeners[type] = listener; }
            };
        },
        body: {
            appendChild: function(el) {
                // MAGIA: Si MediaPipe inyecta un script, lo interceptamos y ejecutamos en RAM
                if (el.tagName === 'script' && el.src) {
                    fetch(el.src)
                        .then(res => res.text())
                        .then(code => {
                            const globalEval = eval;
                            globalEval(code); // Inyectamos MediaPipe en el worker
                            // Simulamos que el script cargó para que la librería continúe
                            if (el.listeners['load']) el.listeners['load'](); 
                        });
                }
            }
        }
    };
}

Con esto ya podía pedir la baja por esquizofrenia, pero eso no es lo importante. Lo importante es que ahora sí, el hilo principal podría pintar tu video, sacar los bytes del fotograma en un "OffscreenCanvas" y dárselos al worker para que te las devuelva sin que la interfaz se congele.

Optimización:

A todo esto hay que añadirle una faceta más, la optimización. Si alguien cree que meter el modelo en un web worker es suficiente para que funcione en un móvil de gama baja, o una de dos: o no tiene ni idea, o para él un móvil de gama baja tiene máximo 3 años. Así que se le añadió por un lado un sistema de "Ping-Pong" donde al modelo se le envía un fotograma, y hasta que no termine de procesarlo no se le envía el siguiente, junto con una "reducción de calidad". El propio modelo reduce el video a 256x256, por lo que mandarle un video en 4k sería una tontería. Es por eso que el video que el modelo analizaba estaba en 480p, haciendo que la carga de reducción fuera menor, y lo más importante, se redujo el número de fps a 15, aunque esto lleva una trampa. Si le muestras a un usuario un video de 60fps con un esqueleto que solo sale cada 4 frames, el usuario notará que el esqueleto va a tirones, y de aquí viene la siguiente sección.

Premium Smoothing y Tempos

Como los videos iban a 15fps, se interpolaron los frames matemáticamente para llegar a los 60fps, aunque eso provocaria que, junto a la interpolación y el propio modelo, que debido a limitaciones de la arquitectura provoca que los puntos tiemblen de un frame a otro, el esqueleto se moviese de forma errática. Así que en vez de comerme esos datos crudos, los pasé por una clase que creé para ello, la clase `BiomechanicsUtils`, que aplica un suavizado matemático para que la curva de movimiento sea perfecta. El resultado es que el esqueleto se moviese con un input lag de 2-3 frames (lo cual ya es una victoria respecto a los 4 iniciales) y con una fluidez y estabilización brutal.

Otros algoritmos

Además de calcular los ángulos de las articulaciones, para calcular la nota y dar datos importantes de cada ejercicio se implementaron algoritmos que detectan el frame (usando los 15 originales para no ensuciar los datos) donde se produce el sticking point, el tempo, profundidad, y otros datos que varían según el ejercicio.

// sw.js (Extracto de programación de Alarmas Offline)

async function programarAlarma(payload) {
    const [hours, minutes] = payload.time.split(':').map(Number);
    const now = new Date();
    let alarmTime = new Date(now.getFullYear(), now.getMonth(), now.getDate(), hours, minutes, 0);
    
    // Si la hora ya pasó hoy, lo programamos para mañana
    if (alarmTime.getTime() <= now.getTime()) {
        alarmTime.setDate(alarmTime.getDate() + 1);
    }

    // Comprobamos si el navegador soporta notificaciones programadas
    if ('showTrigger' in Notification.prototype) {
        try {
            await self.registration.showNotification(payload.label, {
                body: `⏱️ ${payload.time}`,
                icon: '/assets/icons/icon-192.png',
                tag: payload.id, 
                // MAGIA: El SO lanzará la notificación en esta fecha exacta
                showTrigger: new TimestampTrigger(alarmTime.getTime()) 
            });
        } catch (error) {
            console.error('Error al programar la notificación:', error);
        }
    }
}

El infierno del Big Data en el cliente.

Nadie quiere mudarse de app cuando lleva 7 años usando otra, por eso mismo tuve que implementar un sistema de importación y "parser" (que no sé como traducirlo en español ahora mismo, es como traducir datos de un formato a otro) para pasar los datos de apps como Apple Health, Google Fit, Hevy, Strong, etc... a un idioma que mi app sí entiende. Pero claro, como siempre Apple está para joderme la vida. Cuando le pedí a mi amigo que me exportara sus datos para probar el sistema me mandó un archivo XML de 5 GB. Ya no es por el peso (que también tiene lo suyo 5 GB eh, a saber que está guardando ahí y desde hace cuándo), sino que el DOMParser del navegador prefiere suicidarse antes que intentar leerlo.

Por lo tanto, la única solución a semejante locura era leer el archivo en trozos (chunks) de 1 MB junto al uso de expresiones regulares para cazar los datos que mi app sí quiere al vuelo, y si el archivo corta una etiqueta a la mitad entre dos bloques, guarda la parte rota y la concatena con la siguiente lectura.

JAVASCRIPT
// src/modules/onboarding/services/health-parser.service.js
// Extracto: Lectura de archivos XML de varios Gigabytes sin crashear el DOM

const CHUNK_SIZE = 1024 * 1024; // 1 MB
let offset = 0;
let leftover = '';

const readNextChunk = () => {
    // 1. Extraemos un bloque (slice) del archivo gigante sin cargarlo todo
    const slice = file.slice(offset, offset + CHUNK_SIZE);
    reader.readAsText(slice);
};

reader.onload = (e) => {
    // 2. Concatenamos lo que haya sobrado del bloque anterior con el nuevo
    const chunk = leftover + e.target.result;
    let lastIndex = 0;
    
    // 3. Regex para encontrar el nodo, evitando usar el pesado DOMParser
    const recordRegex = /<Record([^>]+)>/g;
    let match;

    while ((match = recordRegex.exec(chunk)) !== null) {
        const attrs = match[2];
        // ... (Lógica de extracción de tipo, valor y fecha con Regex) ...
        
        lastIndex = match.index + match.length;
    }

    // 4. Guardamos la parte del string que no completó un tag XML
    leftover = chunk.slice(lastIndex);
    offset += CHUNK_SIZE;

    // 5. Continuar si no hemos llegado al final
    if (offset < file.size) {
        readNextChunk();
    } else {
        // Finalizado
    }
};

readNextChunk(); // Iniciar proceso

I18N y el modo Tsundere

Cargar todos los idiomas de golpe es una atrocidad para el tiempo de carga inicial, así que Aerko usa importaciones dinámicas asíncronas. Carga solo el diccionario de la pantalla que estás viendo en ese momento y lo cachea. Como la app detecta el idioma del navegador, no hace falta cambiarlo manualmente, pero en el caso de que haya ocurrido un error o simplemente querer cambiarlo porque sí, desde los ajustes puedes activar el idioma que más quieras.

Pero no me quedé únicamente en traducir la app, sino en inyectar personalidades. El sistema soporta "Modos" que cambian tanto los textos como la experiencia. Entre ellos tenemos el default, el modo zen, que ofrece una experiencia de usuario más relajada y comentarios amables, o el modo tsundere, cuyo nombre podría considerarse clickbait dado que el arquetipo tsundere se basa en ocultar el cariño/sentimientos insultando o tratando mal, pero en mi app este modo directamente te odia.

También soporta arrays de textos (varias opciones de las cuales se elige una aleatoriamente) para que la app no te insulte siempre con la misma frase, y variables en crudo con RegExp sobre la marcha (ej: "Solo has levantado `{{weight}}` kg, patético"). (Si alguien pregunta por qué una persona en su sano juicio quiere que su app le insulte, no conozco la respuesta, soy diseñador no psicólogo)

Y como me encantan las coñas, además de un botón que pone "No pulses aquí" el cual te lleva a un rick roll si lo pulsas, también hay una colección de insultos para ver cuántos insultos has desbloqueado.

JAVASCRIPT
// src/core/i18n/i18n.service.js (Extracto simplificado)
async loadPage(pageName) {
    // 1. Cálculo dinámico de rutas basado en idioma y modo
    const basePath = `/src/core/i18n/${this._currentLang}/${pageName}.js`;
    const modePath = this._modeSuffix ? 
        `/src/core/i18n/${this._currentLang}${this._modeSuffix}/${pageName}.js` : basePath;

    // 2. Carga asíncrona (Code Splitting) y fusión (Override)
    let baseData = (await import(basePath).catch(() => ({}))).default || {};
    let modeData = modePath !== basePath ? 
        (await import(modePath).catch(() => ({}))).default || {} : {};
    
    const mergedData = { ...baseData, ...modeData };

    // 3. Wrapper de traducción con variables y arrays aleatorios
    const dictWrapper = {
        data: mergedData,
        t: (key, vars = {}) => {
            let text = mergedData[key];
            
            // Resolución de arrays aleatorios
            if (Array.isArray(text)) {
                text = text[Math.floor(Math.random() * text.length)];
            }
            
            // Inyección de variables con Regex
            if (vars && typeof text === 'string') {
                for (const [vKey, vValue] of Object.entries(vars)) {
                    text = text.replace(new RegExp(`{{${vKey}}}`, 'g'), vValue);
                }
            }
            return text || `[${key}]`;
        }
    };
    return dictWrapper;
}

El "vendor lock" me parece un cáncer, así que la app ofrece la posibilidad de exportar todos tus datos (por separado o todos juntos) en un JSON. Ahora, ¿Qué pasa si tenemos las bóvedas encriptadas? ¿Me exporta la dieta en hebreo? Aunque lo pensé (ironía) la app tiene una función que entra en las bóvedas encriptadas, coge tus registros, los pasa por la llave maestra (la que dije que solo vive en la RAM), los descifra, monta el JSON usando un Blob y te lo descarga. Y ya eres libre de hacer lo que quieras con ellos.

JAVASCRIPT
// src/modules/settings/services/export.service.js
// Extracto: Desencriptación al vuelo y empaquetado de datos en el cliente

async exportData(selectedVaults) {
    const exportPayload = {};

    for (const vaultName of selectedVaults) {
        const rawRecords = await db.getAll(vaultName);
        
        // 1. Si la bóveda está protegida, desencriptamos registro a registro en RAM
        if (isSecurityEnabled && protectedVaults.includes(vaultName)) {
            if (!cryptoService.key) throw new Error("err_export_no_key");
            
            const decryptedRecords = [];
            for (const record of rawRecords) {
                const decryptedData = await cryptoService.decrypt(record.data);
                decryptedRecords.push({ id: record.id, data: decryptedData });
            }
            exportPayload[vaultName] = decryptedRecords;
        } else {
            exportPayload[vaultName] = rawRecords;
        }
    }

    // 2. Generación del archivo JSON local sin usar backend
    const jsonString = JSON.stringify(exportPayload, null, 2);
    const blob = new Blob([jsonString], { type: 'application/json' });
    const url = URL.createObjectURL(blob);
    
    // 3. Forzamos la descarga nativa en el dispositivo
    const a = document.createElement('a');
    a.href = url;
    a.download = `Aerko_Export_${new Date().toISOString().split('T')}.json`;
    document.body.appendChild(a);
    a.click();
    
    // Limpieza de memoria
    setTimeout(() => {
        document.body.removeChild(a);
        URL.revokeObjectURL(url);
    }, 150);
    
    return true;
}
