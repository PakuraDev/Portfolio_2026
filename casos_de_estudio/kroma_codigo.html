1. Arquitectura Vanilla y el EventBus

Si habéis leido otro de mis casos de estudio ya deberíais saber que yo y los frameworks (en especial Angular, que no me voy a cansar de decirlo) no nos llevamos bien, por lo que ya podréis imaginar que Kroma está hecha en Vanilla JS, html y css puro (porque tampoco me gusta Tailwind). La arquitectura base arranca de un main.js que obliga a resolver secuencias de promesas lógicas antes de hidratar el DOM (traducción al castellano para gente que no se odie a si misma: Es como un restaurante donde el camarero, que se llama main.js, no te deja pasar al comedor hasta que la cocina confirma que tiene la comida lista, y en este caso has pedido una de promesas. Una vez todo está listo, te sirven la mesa, o en este caso, hidratan el DOM).

Ahora, para que esto funcione bien sin tener un jeroglífico en JS, nuevamente enchufo un bus de eventos con el patrón Publisher/Subscriber. Al igual que en Aerko_, el bus de eventos nos permite aislar las ejecuciones en bloques try-catch. Es decir, si hay un módulo que no funciona, lo unico que no funciona es ese módulo, pero el resto de la web si. Os dejo por aqui un pequeño snippet de como funciona:

JAVASCRIPT
// JS/core/bus.js
emit(eventName, data) {
    if (!this.listeners[eventName]) return;

    // Clonamos el array porque sino los bus.off() anidados la lían
    const handlers = [...this.listeners[eventName]];
    handlers.forEach(callback => {
        try {
            callback(data);
        } catch (error) {
            // Evitamos un fatal error en cadena
            console.error(`[EventBus] Error in listener for '${eventName}':`, error);
        }
    });
}

2. Bóvedas Locales y Promesas (KromaDB)

Cualquier persona normal hubiera usado localStorage para los datos que guarda el sistema, pero yo no soy normal, soy idiota. Si alguien lo ha notado ya, obviamente reutilicé un poco de la lógica base de aerko, aunque adaptada, entonces claro, podía volver a usar un wrapper de IndexedDB para asegurarme de que, aunque el cliente sea un esquizofrenico con 9000 enlaces, el sistema no se quede sin espacio (porque localStorage tiene límite de 5mb).

Así que hice que la verificación del esquema con "onupgradeneeded" fuerce claves primarias (keypath, id, etc...), diviendiendo los esquemas en 5 stores, "user", "groups", "Categories", "links" y "notepads". Fun fact, si le preguntaís a una IA sobre la arquitectura de datos os propone solo 2 stores, uno para los enlaces y otro para el notepad. Posible es y dentro de lo que cabe tampoco sería mala con un array, pero para volverse loco con un solo bug también funcionaba.

JAVASCRIPT
// JS/core/db.js
async put(storeName, item) {
    await this.ready; // Retenemos ejecuciones en crudo garantizando el I/O
    return new Promise((resolve, reject) => {
        const transaction = this.db.transaction([storeName], 'readwrite');
        const store = transaction.objectStore(storeName);
        const request = store.put(item);

        request.onsuccess = () => resolve();
        request.onerror = (e) => reject(e.target.error);
    });
}

3. El Cerebro: CLI y Regex

Aquí es donde el proyecto destaca, porque claro, ¿Cuántas páginas de inicios existen? Pues solo en github hay casi 5000 proyectos, así que hacer otro solo por diseño sería una perdida de tiempo. Es por eso que le dí un pequeño toque de linux he hice que la barra de búsqueda sea un CLI (Command Line Interface) personalizado y optimizado para lo que hago. Además de funcionar como una barra de busqueda normal y corriente, también sirve para ejecutar comandos como:

  • "/yt [busqueda]" -> Busca en youtube. Me ahorra el tener que entrar en youtube y usar la barra de busqueda, aquí directamente busco lo que quiero y me lleva a la busqueda.
  • "/hex [color en hex]" -> Te lleva a encycolorpedia.es, la biblia de los colores.
  • "/svg [busqueda]" -> Busca en svgrepo.com. Que por cierto, me dí cuenta el otro día que usan vercel. Dentro de lo que cabe lo entiendo, pero se podría haber hecho con cloudflare, edge computing, wasm y cliente.
  • "/roll [nº máximo]" -> Tira un dado. Ya está, así de tonto.
  • "/choose [opciones]" -> Elige una opción entre varias. Por ejemplo: "/choose Er betih, barça, vardrid" y automaticamente te elige el betis porque el código tiene arte, pero si lo pones entre otras cosas si te sale uno aleatorio.
  • "/export" -> Exporta los datos
  • "/import" -> Importa los datos

Además de los comandos, también te permite hacer operaciones matemáticas nativamente. Podría haber usado una librería, pero preferí blindar la lógica con expresiones regulares, entre otras cosas porque la cuenta más compleja que hago es "(1440-120)/3". Eso si, os recomiendo mathway para cosas más complejas, esa página me ayudó en 1 bach.

JAVASCRIPT
// JS/logic/searchEngine.js
// Parseo y securización del input crudo
const isMath = /^[0-9+\-*/().\s]+$/.test(query) && /[0-9]/.test(query);
const isOperator = /[+\-*/]/.test(query); 
        
if (isMath && isOperator) {
    try {
        const result = new Function(`return ${query}`)();
        if (result !== undefined && !isNaN(result) && result !== Infinity) {
            const finalResult = Number.isInteger(result) ? result : parseFloat(result.toFixed(4));
            // Escupe el cálculo en línea con la acción CLI
            return { action: 'inline', payload: String(finalResult) };
        }
    } catch (e) {
        // En caso de sintaxis incompleta
    }
}

4. Decoradores y SVG Crudo

En vez de limitarme a usar Base64 como un programador perezoso (notese la ironia), separé por completo la estructura MIME en decorators.js. Cuando subes un SVG utilizo FileReader.readAsText() para extraer directamente el vector original y empotrarlo en el DOM. ¿El motivo? Pues que soy gilipo... digo, al inyectar el svg como texto crudo en el html, ese svg hereda propiedades universales nativas como currentColor del theme engine principal. Una buena forma de no volverme loco los 5 estilos del proyecto y los svg.
