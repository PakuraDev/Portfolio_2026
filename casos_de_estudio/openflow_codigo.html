A ver, hablar del código de OpenFlow_V3 es algo que me apetece bastante, porque este proyecto es la encarnación de mi obsesión por el rendimiento extremo, el cero lag y mi odio profundo hacia la latencia de red. Hoy en día parece que si no usas una herramienta corporativa lenta que necesita cargar medio segundo cada vez que mueves una tarjeta, no eres profesional. Pues al carajo con eso. OpenFlow es una SPA Local-First de grado industrial, brutalista y cruda.

Para no volveros locos (ni volverme loco yo), voy a dividir este caos en 4 partes:

1. Arquitectura Local-First y Soberanía de Datos (IndexedDB)

Si vienes de leer mis otros casos de estudio, ya sabéis que yo y los frameworks (cof cof, Angular, cof cof) somos enemigos naturales. OpenFlow_V3 está construido en Vanilla JS puro y duro. Pero la verdadera joya aquí no es que no use React o parecidos, sino que no hay servidor.

En una web tradicional (tipo Jira), tú mueves una tarea, tu ordenador le manda un mensajito al servidor (que a saber dónde está alojado), el servidor lo procesa, lo guarda, y te responde "vale, ya está". Todo eso tarda unos milisegundos (o segundos si tu internet es el de Movistar un domingo lloviendo). En OpenFlow, tu navegador es el servidor. Usamos la base de datos nativa del propio navegador: IndexedDB.

IndexedDB es brutalmente rápido, pero su API nativa te da ganas de llorar por el horroroso sistema de callbacks. Así que, con la filosofía de un artesano (o masoquista), me monté mi propio wrapper con promesas en db.js para hacer un CRUD genérico asíncrono y meterle un sistema de migraciones (para poder pasar de la v1 a la v2 sin perder datos en el proceso añadiendo épicas e inyectando fechas).

La mayor ventaja del enfoque Local-First es la "Soberanía de Datos". Tus proyectos residen en tu máquina y de ahí no salen salvo que tú exportes. No hay tracking corporativo ni intermediarios. Y lo más importante: la latencia real es literalmente cero. Cero "loading spinners". Si se cae el internet, tú sigues trabajando como si de un software escritorio se tratara.

2. El Motor de Rendering Optimista y el Radar de Colisiones

Mover una tarjeta en un tablero Kanban parece una tontería de lo más sencilla, pero hacerlo fluido a 60fps sin que el navegador estornude tiene su truco. Aquí entra lo que yo llamo "Optimistic UI Rendering". Cuando tú arrastras una tarjeta y la sueltas, la interfaz gráfica se actualiza de forma instantánea en el DOM (en apenas 0ms), mientras que el cambio en sí de estado se manda a guardar a la base de datos de fondo, en total asincronía. Es silencioso pero letal.

Pero claro, la API nativa de Drag & Drop de HTML5 es todo un dolor de cabeza. Principalmente porque el navegador no tiene un sentido claro de dónde quieres soltar exactamente la tarjeta en medio de una columna (queremos ordenación manual). Así que le programé su propio "Radar".

JAVASCRIPT
// js/ui/render.js (Extracto de la magia matemática del radar)
function getDragAfterElement(container, y) {
    const draggableElements = [...container.querySelectorAll('.task-card:not(.dragging)')];
    return draggableElements.reduce((closest, child) => {
        const box = child.getBoundingClientRect();
        const offset = y - box.top - box.height / 2; // Distancia al centro de la tarjeta
        if (offset < 0 && offset > closest.offset) {
            return { offset: offset, element: child };
        } else {
            return closest;
        }
    }, { offset: Number.NEGATIVE_INFINITY }).element;
}

Básicamente, es una función matemática que calcula dónde está el ratón comparado con el centro vertical del resto de cartas de la columna para averiguar a ciencia exacta debajo de cuál la quieres meter. Una vez el radar hace su trabajo, entra en escena la versión pro de la lógica de soltado (la "The Benito Villamarín Edition"): el script pilla todas las cartas de la lista, les otorga un nuevo valor order por su índice actual y procede con el guardado, de modo que el ranking se conserva inmutable al refrescar.

3. Delegación de Eventos Estricta (El Guardia de Seguridad)

Si alguna vez una Single Page Application (SPA) se te empieza a calar al llevar media hora trabajando, lo más probable es que tenga "Memory Leaks" (fugas de memoria). Si renderizas 200 tarjetas, y a cada botoncito que haya le enchufas un addEventListener, estás malgastando la RAM y creando basura computacional. Y si las sacas del DOM sin desasignar los eventos, ya apaga y vámonos.

Para mantener una arquitectura con un perfil de memoria plano para gente que tiene el OpenFlow abierto 8 horas seguidas, he aplicado una política estricta de Event Delegation global.

Mencionando la lógica de modales (modal.js), en lugar de atarle un evento a cada triste botón para borrar épicas, el modal-body pilla el rol de portero de discoteca, o mejor dicho, de Germán Pezzella cortando un contraataque. Todo clic rebotará hasta él y mediante condicionales e.target.matches() decidirá a quién se lleva por el medio. Un solo guardián, y miles de elementos temporales que nacen y mueren limpios.

Y por supuesto, una anécdota imperdonable en render.js. Para impedir ataques de Cross-Site Scripting (XSS) al inyectar puro texto como HTML, tuve que calzarle una función urgente de sanitización que convierte etiquetas como <script> en inocofensivas entidades &lt;. "Lo siento gente, el que no escape los HTML no es bético". Qué le hacemos, pragmatismo ante el desastre inminente.

4. Domesticando el Cronograma (Frappe) y el "Radar" de Búsqueda

Para el cronograma usé la librería frappe-gantt. Como no iba a reinventar la rueda trazando cuadros en CANVAS, me traje Frappe con un único problema: el de siempre, los datos los espera en un formato, y mi BBDD los tiene en otro muy dispar.

Aquí, para que nadie se me eche encima, entra el todopoderoso Patrón Adapter (gantt.js). Mapeo y transformo al vuelo todas mis tareas de OpenFlow hacia el esqueleto que espera Frappe, con el añadido de interceptar primero la fase (Épica) para robarle su color hexagonal y colarlo en cada barra del cronograma. Y para forzar mi "paleta industrial brutalista" en unos SVG que se resistían... uso ataques de fuerza bruta temporal (o lo que es lo mismo, clavarle los estilos mediante JS tras un pequeño setTimeout).

Y, ya para coronar la corona tecnológica, está mi buscador en tiempo real. ¿Cómo busca otra SPA habitual las cosas? Mandan la string (texto) al backend, hacen una pseudo búsqueda indexada en la BD, se devuelven los resultados por una REST API, recálculo bestial, y se actualiza el DOM de React. ¿En OpenFlow? Simplemente tienes un listener pillando tu tecleo, interceptando los nodos del DOM uno por uno y comparándolos. Todo aquello que no coincida... display: none. Sin red, sin esperas asíncronas, visibilidad oculta temporal, instantáneo y gratis a nivel de computación para la interfaz.

Es un código áspero, libre y reaccionario en todos los sentidos, pero puedo afirmar orgullosamente que la usabilidad resultante justifica cada pequeña locura. Espero que tampoco os haya explotado la cabeza leyendo mi forma de entender la programación, ¡nos vemos en la próxima interfaz!
