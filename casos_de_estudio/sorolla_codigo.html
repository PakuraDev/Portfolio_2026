Explicación rápida

Bien, imaginaos que construir una web es como montar una exposición en un museo (como un museo, y justo la web es un museo, flipad con mi nivel de creatividad). Al hacer la web, podéis tomar dos caminos, o bien os vais a lo seguro y lo decoráis con muebles y accesorios fabricados en serie (un framework), o directamente los fabricáis vosotros mismos a vuestro gusto (JS Vanilla). Por otro lado, podríais pintar todos los cuadros en la pared del museo (un archivo monolítico), lo cual sería un caos para reorganizar, o bien podrías pintar cada cuadro en su propio lienzo, y luego colgarlos en las salas (webcomponents).

Pues así es como está construido esta web. Todo está construido en HTML, CSS puro y JS Vanilla, y modularizado para que cada componente (el header, una sección, un botón, etc...) sea un archivo independiente que se importa y renderiza en la web. Eso sí, esto no es lo óptimo, pero al ser una sola página de ejemplo para mostrar el diseño no tiene mayores implicaciones.

Arquitectura Vanilla y Web Components

La mayoría de webs hoy en día utilizan frameworks como React, Vue o Astro (este último solo si sois gigachads), y quiero decir, es cómodo, rápido, con todo ya construido. Pero mi filosofía de rendimiento excesivo (que ya roza la locura) me impide usarlo, además de que para está web (que ya digo, solo es una página de 5 componentes) me parece hasta sobre-ingeniería (irónico que diga esto cuando podría haber metido todos los componentes en un solo archivo).

Ahora, que vaya a usar JS Vanilla no significa que vaya a meterlo todo en un solo archivo, porque si habéis leído alguno de los readme de mis repositorios sabréis que suelo modularizar absolutamente todo. Por eso mismo, cada sección de la web está hecha con webcomponents y simplemente se importan en el index.html con una clase de JS que extiende HTMLElement y se renderizan con this.innerHTML (Sé que dije que no iba a ser demasiado técnico, pero a ver, es que esto no se puede poner de otra forma) junto etiquetas como "".

JAVASCRIPT
class="code-keyword">export class AgendaSection extends HTMLElement {
  constructor() { super(); }

  connectedCallback() { this.render(); }

  render() {
    this.innerHTML = `
      <style>
        /* CSS encapsulado de la sección */
      </style>
      <section class="agenda-layout">
        </section>
    `;
  }
}
customElements.define('agenda-section', AgendaSection);

CSS & Design Tokens

Aunque el sistema de colores parezca puro diseño, la implementación técnica es, como bien dice su nombre, técnica (espero que esta broma le haya hecho gracia a alguien). Aunque creo que es algo simple, al parecer hay personas y webs que todavía no usan variables, así que voy a presumir de que mi proyecto usa Variables CSS, configuradas como tokens de diseño, en un archivo independiente.

Esto permite que la web "respire" (nada, hoy estoy poético) bajo un sistema lógico: desde los cuatro tonos de "--arena" que imitan la orilla del mar hasta el "--azul-marino" institucional. Es una arquitectura de estilos que evita la repetición de código y permite cambios globales de forma instantánea, manteniendo la coherencia en todos los componentes.

Botones "Líquidos"

Aunque están desactivados debido al shadow dom y arquitectura usada, los botones tienen programado un efecto "Líquido" al hacer hover. Esto, sin librerías externas, se hace con un truquito técnico que combina CSS y filtros SVG: Dentro del index.html definí un filtro global oculto que utiliza "feTurbulence" para generar un ruido fractal animado. Luego, desde el CSS de los componentes, "llamo" a ese filtro mediante su ID cuando el usuario hace hover sobre el botón, aunque como ya digo, el Shadow DOM y Webcomponents no dejan que el efecto se muestre. Igualmente, para no quedar como un mentiroso os dejo un snippet del código:

JAVASCRIPT
/* Filtro definido en el HTML global */
<filter id="subtle-liquid">
  <feTurbulence type="fractalNoise" baseFrequency="0.03 0.1" numOctaves="1">
    <animate attributeName="seed" class="code-keyword">from="1" to="100" dur="12s" repeatCount="indefinite" />
  </feTurbulence>
</filter>

/* Aplicación técnica en el componente */
.btn-sorolla:hover .btn-sorolla-fluid {
  opacity: 1;
  filter: url(#subtle-liquid);
}

Pragmatismo del Desarrollador (o chapuza)

Aquí voy a permitir que todos me miréis mal, pero también quiero que me entendáis. La web no es responsive, entras desde un móvil o una tablet y está rota. ¿La podría haber adaptado? Sí, perfectamente, pero claro, hay que tener en cuenta dos cosas:

1. Nadie me paga por hacer la web: Aquí todos me podéis insultar la verdad, es la excusa más corporativa que hay, pero entendedme, no es un proyecto para un cliente, sino un rediseño no oficial que la gente puede saltarse.

Así que sí, podéis insultarme, podéis llamarme flojo, y podéis decir que soy un chapucero, no lo voy a negar porque en este caso técnicamente es cierto. Si algún día un cliente me pide que la haga (lo que significará que estáis leyendo esto), le echaré unos días extras y hago responsive. Hasta entonces me vais a tener que perdonar.
