Quick Explanation

Alright, to explain everything, let's imagine building an app is like running a theater. Most apps are made using a framework (like React, Angular (if anyone still uses that), or Astro if you're a gigachad), which is basically like going to IKEA and buying furniture with the wood already cut, screws included, and an instruction manual. It's super convenient and fast. But I'm a performance obsessive (or a masochist, both definitions are correct), so I used Vanilla JS, which in this context is exactly the same as getting in the car, going to the forest, making an axe, cutting down a tree, sanding the wood, and building the furniture with my own hands.

If you understood that without needing to watch a tutorial or ask ChatGPT, then you're doing great. Moving on to the DOM, which would basically be the theater stage with its sets. In a traditional website, every time you click a button, the theater closes the curtain and changes the set. But my website is an SPA (Single Page Application), which means the curtain never closes. When you switch tabs, Flash (or anything with inhuman speed you can imagine) changes the set at the speed of light while you keep watching without needing to reload.

Now let's talk about the router, which is more or less like a club bouncer. To manage set changes without anyone sneaking in, I programmed my own router. Every time you try to go to a private section of the app, the router intercepts you and asks "Do you have the crypto session open?" or in Andalusian equivalent "Oi, are you from Betis or Sevilla?" If you've entered the pin (or are a Betis fan) it lets you in. If not, it kicks you out and sends you straight to the tax office for a surprise audit (By the way, let me clarify that I'm not talking about the router in your house, but a small script).

Now the event bus (or EventBus), which I'll call "megaphone." Since I don't go to IKEA, the different parts of the app (furniture) don't come connected to each other, but I need the nutrition section to know that Juan has logged his weight so it can recalculate his calories. So I programmed an event bus, which is basically like a megaphone that shouts what you're doing to the rest of the app.

Now for what's probably the hardest thing to explain: Web Components and the Shadow DOM. Basically, it's like each component of the app is a tupperware container. Let's move from the theater to a bar kitchen. Sometimes the design (CSS) is chaos. Sometimes you have a red button on the home page, but due to a cascading error, it turns red in settings too, when the settings buttons are blue. Explained with the bar, it's like the waiter writes down 3 orders, but the chef gets confused and puts onions on all the dishes when only one was supposed to have them. To avoid this, the waiter gives the 3 tickets separately, and the chef puts each meal in an airtight container so the ingredients don't mix. Each container is a web component, and the shadow DOM is the food inside.

And with these basic concepts, now comes the exam. I'm going to start talking like a nerd, so if you don't understand what follows, don't worry — that's normal.

The Guardian Router

Making a custom router based on window.location.hash is easy. The hard part is making that router a security paranoid. In Aerko_, before rendering a single element, the router intercepts the route, asks the authentication service if the crypto vault is open (authService.hasPin()), and routes to the private section or login accordingly. It also handles the user lifecycle — if you're a new user, it kidnaps... I mean, kindly takes you to onboarding. All this by injecting raw HTML and/or classes with replaceChildren() so the screen doesn't even flicker.

EventBus

As a proper gigachad who doesn't go to IKEA, I had to build a native event bus based on the Publisher/Subscriber pattern (In human terms, components don't talk to each other directly; they shout to a common room and whoever's interested listens). I also added a debug mode that prints all events in the browser console so I don't go crazy tracking data, although I went crazy anyway building it.

Almost all (if not all) of the UI is built with native Web Components. And as you can imagine, most use shadow DOM, although there was a point in development where I went for light DOM (which is the opposite of shadow DOM), so in this case it's about 60-80% shadow DOM and 20-40% light DOM.

Data Persistence and Cryptography (Privacy by Design)

This is where things get serious. Your phone (or whatever device you use the app on — maybe someone uses it on an iPad or even a mini PC, it's possible) is the database, which means if someone steals it or plugs it into a PC (knowing your PIN), your data could be vulnerable. Although it's optional (since the data isn't sensitive), I built an end-to-end encrypted vault architecture using IndexedDB and the Web Crypto API. If you activate it, your data will be stronger than vinegar, while if you don't, it's vulnerable (though honestly, I don't think anyone cares if their daily calories get leaked).

Vaults and the Transparent Wrapper

I'm not gonna lie, whoever made IndexedDB must have hated their job. It's literally unbearable with its callback system, so I directly made a wrapper class with promises to make it transactional (I also considered using sqlite in wasm, but since I've worked with IndexedDB more often, I went with the safe option). Obviously I divided the database into different vaults, like "Public_store" (for silly things like language or theme) and armored vaults like "user_vault", "training_vault", etc.

The fun part is the DX (Developer Experience). The rest of the app's code doesn't know about encryption — it just tells the database "Oi, can you save this for me?" and the database class intercepts "this" (the object), checks if it's going to a protected vault, and if so, generates a unique 12-byte initialization vector (IV) that gets encrypted with AES-GCM on the fly. It's been very nerdy, but the result is basically going from "I ate potatoes" to "iv: new Uint8Array([122, 201, 8, 45, 19, 110, 2, a long etc that I'm not going to write])". To show you the level of interception I'm talking about, here's a snippet from the code:

JAVASCRIPT
// src/core/db/index.js (Extract of interception)
async put(storeName, data) {
    // 1. Bypass if the vault is public or there's no key in RAM
    if (!this._isProtected(storeName) || !cryptoService.key) {
        return this._transaction(storeName, 'readwrite', store => store.put(data));
    }

    // 2. Interception and on-the-fly encryption (AES-GCM)
    const { id, ...payload } = data;
    const encryptedData = await cryptoService.encrypt(payload);

    const recordToSave = {
        id: id,
        data: encryptedData // Only the IV and encrypted buffer are saved
    };

    // 3. Secure insertion
    return this._transaction(storeName, 'readwrite', store => store.put(recordToSave));
}

RAM Memory and Paranoia

Now, where does the key to encrypt that come from? From the PIN you enter when logging in. Using PBKDF2 (which although I would've liked to use Argon2, that seemed totally excessive, plus it would've been a wasm and complicated everything, since Argon2 is a hash), 100,000 iterations, and a random "Salt", an AES-GCM key is derived that would make any Russian hacker cry. And here's the genius (more like basic security xd): that key only lives in RAM. Every time you lock or exit the app, this.key=null runs and the key disappears. Without your PIN, it's mathematically impossible. So much so that the system is close to being quantum-proof.

Finally, to check if the PIN is correct without saving the actual PIN (which would be a symptom of severe idiocy), the "Canary" pattern is used. I store an encrypted object that says "{ secret: 'AERKO_SECURE' }", and when the user enters their PIN, it tries to decrypt the canary. If it returns 'AERKO_SECURE', I know it's you and I let you in. If not, the canary dies and you stay out. This isn't bulletproof either — obviously anyone bored enough could try brute force, but I don't think anyone would find it worth it to steal a phone, extract the app data (which is in the browser, making it even harder to export), decrypt it via brute force, all just to find out your leg day routine (or whether you actually skip it — I've caught you).

Web Workers and Offline Capability

Following my philosophy of individual freedom, and to allow any device to use the feature, I couldn't just include a single model and force people to use it. So I implemented a download system where you can choose between Lite, Full, and Heavy (along with a recommendation system) and save it in the browser cache. But there's a small problem: If I put an AI (which is already heavy) analyzing a video on the same thread that processes the screen, the app freezes. The solution? Put the AI in a Web Worker so it works in the background.

But wait, if you thought my misery ended here, you haven't used a library made by Google. Mediapipe assumes it's on a normal page (which I can't really blame it for) and looks for objects like "window" or "document", but of course, in the Web Worker those objects don't exist. So I had no choice but to create a fake DOM so the AI thinks it's on a normal page. Let me drop a small extract here so you can see the level of insanity:

JAVASCRIPT
// src/modules/training_lab/views/pose-worker.js (Extract of interception)

// TRICK: The fake 'document' (Expert level with script interceptor)
if (typeof document === 'undefined') {
    self.document = {
        createElement: function(tag) {
            // If the AI tries to create a canvas, we give it an OffscreenCanvas
            if (tag === 'canvas') {
                const canvas = new OffscreenCanvas(1, 1);
                canvas.setAttribute = function() {};
                if (!canvas.addEventListener) canvas.addEventListener = function() {};
                return canvas;
            }

            // We create a fake object for any other element (e.g., scripts)
            return {
                tagName: tag.toLowerCase(),
                src: '',
                listeners: {},
                setAttribute: function(name, value) { if (name === 'src') this.src = value; },
                addEventListener: function(type, listener) { this.listeners[type] = listener; }
            };
        },
        body: {
            appendChild: function(el) {
                // MAGIC: If MediaPipe injects a script, we intercept and execute it in RAM
                if (el.tagName === 'script' && el.src) {
                    fetch(el.src)
                        .then(res => res.text())
                        .then(code => {
                            const globalEval = eval;
                            globalEval(code); // Inject MediaPipe into the worker
                            // Simulate that the script loaded so the library continues
                            if (el.listeners['load']) el.listeners['load']();
                        });
                }
            }
        }
    };
}

With this I could already file for schizophrenia leave, but that's not the important part. The important thing is that now the main thread could paint your video, extract the frame bytes into an "OffscreenCanvas" and give them to the worker for processing without the interface freezing.

Optimization:

On top of all this, there's another facet: optimization. If anyone thinks putting the model in a web worker is enough to run on a low-end phone, either they have no idea, or to them a low-end phone is at most 3 years old. So I added a "Ping-Pong" system where a frame is sent to the model, and until it finishes processing it, the next one isn't sent, along with a "quality reduction." The model itself reduces the video to 256x256, so sending a 4k video would be stupid. That's why the video the model analyzed was in 480p, making the reduction load smaller, and most importantly, the fps was reduced to 15, although this comes with a catch. If you show a user a 60fps video with a skeleton that only updates every 4 frames, the user will notice the skeleton is jerky — which leads to the next section.

Premium Smoothing and Tempos

Since the videos ran at 15fps, the frames were mathematically interpolated to reach 60fps, although that would cause — along with the interpolation and the model itself, which due to architecture limitations causes points to shake between frames — the skeleton to move erratically. So instead of feeding those raw data points, I ran them through a class I created for this purpose, the `BiomechanicsUtils` class, which applies mathematical smoothing so the movement curve is perfect. The result is that the skeleton moved with an input lag of 2-3 frames (which is already a victory compared to the initial 4) with brutal smoothness and stabilization.

Other Algorithms

In addition to calculating joint angles, to compute the score and provide important data for each exercise, algorithms were implemented that detect the frame (using the original 15 to avoid dirtying the data) where the sticking point occurs, the tempo, depth, and other data that varies depending on the exercise.

Service Workers and Serverless Alarms

For those who don't know, Aerko is a PWA (Progressive Web App, or in plain English, an installable web application), so we need a Service Worker. In this situation I had to choose between making a "normal" Service Worker, an offline one, or being me and making a Service Worker that acts like the tax office handling the network (if the network were my taxes) and sending notifications (also like the tax office) without a server.

If there's no coverage, the app works just as well, but the crown jewel is the notifications, which unfortunately won't work in the ecosystem until Apple deigns to stop hating iOS users. But to the point: to send you a notification reminding you it's time to eat at 14:30 without using a backend, I used the experimental "Notification Triggers" API (which, as you can guess, doesn't work very well on either Android or Apple).

JAVASCRIPT
// sw.js (Extract of Offline Alarm scheduling)

async function programarAlarma(payload) {
    const [hours, minutes] = payload.time.split(':').map(Number);
    const now = new Date();
    let alarmTime = new Date(now.getFullYear(), now.getMonth(), now.getDate(), hours, minutes, 0);

    // If the time has already passed today, schedule for tomorrow
    if (alarmTime.getTime() <= now.getTime()) {
        alarmTime.setDate(alarmTime.getDate() + 1);
    }

    // Check if the browser supports scheduled notifications
    if ('showTrigger' in Notification.prototype) {
        try {
            await self.registration.showNotification(payload.label, {
                body: `⏱️ ${payload.time}`,
                icon: '/assets/icons/icon-192.png',
                tag: payload.id,
                // MAGIC: The OS will fire the notification at this exact date
                showTrigger: new TimestampTrigger(alarmTime.getTime())
            });
        } catch (error) {
            console.error('Error scheduling notification:', error);
        }
    }
}

The Hell of Big Data on the Client

Nobody wants to switch apps when they've been using another one for 7 years, so I had to implement an import and parsing system to translate data from apps like Apple Health, Google Fit, Hevy, Strong, etc. into a language my app can understand. But of course, as always, Apple is there to ruin my life. When I asked my friend to export their data for me to test the system, they sent me a 5 GB XML file. It's not just the file size (though 5 GB is no joke — who knows what they've been saving and since when), but the browser's DOMParser would rather commit suicide than try to read it.

So the only solution to such madness was to read the file in chunks of 1 MB using regular expressions to grab the data my app wants on the fly. And if the file cuts a tag in half between two blocks, it saves the broken part and concatenates it with the next read.

JAVASCRIPT
// src/modules/onboarding/services/health-parser.service.js
// Extract: Reading multi-gigabyte XML files without crashing the DOM

const CHUNK_SIZE = 1024 * 1024; // 1 MB
let offset = 0;
let leftover = '';

const readNextChunk = () => {
    // 1. Extract a block (slice) from the giant file without loading it all
    const slice = file.slice(offset, offset + CHUNK_SIZE);
    reader.readAsText(slice);
};

reader.onload = (e) => {
    // 2. Concatenate anything left over from the previous block with the new one
    const chunk = leftover + e.target.result;
    let lastIndex = 0;

    // 3. Regex to find the node, avoiding the heavy DOMParser
    const recordRegex = /<Record([^>]+)>/g;
    let match;

    while ((match = recordRegex.exec(chunk)) !== null) {
        const attrs = match[2];
        // ... (Extraction logic for type, value, and date with Regex) ...

        lastIndex = match.index + match.length;
    }

    // 4. Save the part of the string that didn't complete an XML tag
    leftover = chunk.slice(lastIndex);
    offset += CHUNK_SIZE;

    // 5. Continue if we haven't reached the end
    if (offset < file.size) {
        readNextChunk();
    } else {
        // Finished
    }
};

readNextChunk(); // Start process

If you're coming from Google Fit (which exports as .zip), there's a problem. Do I force the user to give me the exact file? It was viable since they're supposed to be advanced users, but let's be honest, if my professor Billy saw me allowing that, he'd fail me on the spot. So I added JSZip to unzip the file in the browser's RAM asynchronously, find the .csv containing the word "weight", extract the text, and parse it.

On the other hand, for workout apps like Hevy there was an even bigger problem. If my app has bench press as "Press_Banca" but Strong has it as "Press-Banca", it would be impossible to parse the data. So I had to make a heuristic engine that takes their "Press-Banca", cleans it, cross-references it with an internal dictionary, and maps it to "Press_Banca" (I could have also done fuzzy search with fuse.js, but that would add another external library with a chance of interpreting "incline bench press" as "bench press").

I18N and Tsundere Mode

Loading all languages at once is an atrocity for initial load time, so Aerko uses dynamic async imports. It only loads the dictionary for the screen you're currently viewing and caches it. Since the app detects the browser language, there's no need to change it manually, but in case of an error or if you just want to change it anyway, you can select any language from settings.

But I didn't stop at just translating the app — I injected personalities. The system supports "Modes" that change both the text and the experience. Among them we have default, zen mode, which offers a more relaxed user experience with friendly comments, and tsundere mode, whose name could be considered clickbait since the tsundere archetype is based on hiding affection/feelings by insulting or treating someone badly, but in my app this mode just straight up hates you.

It also supports arrays of texts (multiple options from which one is chosen randomly) so the app doesn't always insult you with the same phrase, and raw variables with RegExp on the fly (e.g., "You only lifted `{{weight}}` kg, pathetic"). (If anyone asks why a person in their right mind would want their app to insult them, I don't know the answer — I'm a designer, not a psychologist)

And because I love jokes, besides a button that says "Don't press here" which takes you to a Rick Roll if you press it, there's also a collection of insults to see how many insults you've unlocked.

JAVASCRIPT
// src/core/i18n/i18n.service.js (Simplified extract)
async loadPage(pageName) {
    // 1. Dynamic route calculation based on language and mode
    const basePath = `/src/core/i18n/${this._currentLang}/${pageName}.js`;
    const modePath = this._modeSuffix ?
        `/src/core/i18n/${this._currentLang}${this._modeSuffix}/${pageName}.js` : basePath;

    // 2. Async loading (Code Splitting) and merging (Override)
    let baseData = (await import(basePath).catch(() => ({}))).default || {};
    let modeData = modePath !== basePath ?
        (await import(modePath).catch(() => ({}))).default || {} : {};

    const mergedData = { ...baseData, ...modeData };

    // 3. Translation wrapper with variables and random arrays
    const dictWrapper = {
        data: mergedData,
        t: (key, vars = {}) => {
            let text = mergedData[key];

            // Random array resolution
            if (Array.isArray(text)) {
                text = text[Math.floor(Math.random() * text.length)];
            }

            // Variable injection with Regex
            if (vars && typeof text === 'string') {
                for (const [vKey, vValue] of Object.entries(vars)) {
                    text = text.replace(new RegExp(`{{${vKey}}}`, 'g'), vValue);
                }
            }
            return text || `[${key}]`;
        }
    };
    return dictWrapper;
}

"Vendor lock" seems like cancer to me, so the app offers the ability to export all your data (separately or all together) in JSON. Now, what happens if we have encrypted vaults? Would it export the diet in Hebrew? Although I considered it (sarcasm), the app has a function that enters the encrypted vaults, grabs your records, runs them through the master key (the one that only lives in RAM), decrypts them, builds the JSON using a Blob, and downloads it. And you're free to do whatever you want with them.

JAVASCRIPT
// src/modules/settings/services/export.service.js
// Extract: On-the-fly decryption and data packaging on the client

async exportData(selectedVaults) {
    const exportPayload = {};

    for (const vaultName of selectedVaults) {
        const rawRecords = await db.getAll(vaultName);

        // 1. If the vault is protected, decrypt record by record in RAM
        if (isSecurityEnabled && protectedVaults.includes(vaultName)) {
            if (!cryptoService.key) throw new Error("err_export_no_key");

            const decryptedRecords = [];
            for (const record of rawRecords) {
                const decryptedData = await cryptoService.decrypt(record.data);
                decryptedRecords.push({ id: record.id, data: decryptedData });
            }
            exportPayload[vaultName] = decryptedRecords;
        } else {
            exportPayload[vaultName] = rawRecords;
        }
    }

    // 2. Generate local JSON file without using backend
    const jsonString = JSON.stringify(exportPayload, null, 2);
    const blob = new Blob([jsonString], { type: 'application/json' });
    const url = URL.createObjectURL(blob);

    // 3. Force native download on the device
    const a = document.createElement('a');
    a.href = url;
    a.download = `Aerko_Export_${new Date().toISOString().split('T')}.json`;
    document.body.appendChild(a);
    a.click();

    // Memory cleanup
    setTimeout(() => {
        document.body.removeChild(a);
        URL.revokeObjectURL(url);
    }, 150);

    return true;
}
