1. Vanilla Architecture and the EventBus

If you've read any of my other case studies, you should already know that frameworks (especially Angular, I'm not going to get tired of saying it) and I don't get along, so you can already imagine that Kroma is made with Vanilla JS, HTML, and pure CSS (because I don't like Tailwind either). The base architecture starts from a main.js that forces resolution of logical promise sequences before hydrating the DOM (translation for people who don't hate themselves: It's like a restaurant where the waiter, named main.js, won't let you into the dining room until the kitchen confirms the food is ready, and in this case you ordered a sequence of promises. Once everything's ready, they serve the table, or in this case, hydrate the DOM).

Now, to make this work well without ending up with a JS hieroglyph, I plug in an event bus with the Publisher/Subscriber pattern again. Just like in Aerko_, the event bus allows us to isolate executions in try-catch blocks. That is, if a module fails, only that module fails, but the rest of the website works. Let me leave a small snippet of how it works:

JAVASCRIPT
// JS/core/bus.js
emit(eventName, data) {
    if (!this.listeners[eventName]) return;

    // We clone the array because otherwise nested bus.off() calls mess things up
    const handlers = [...this.listeners[eventName]];
    handlers.forEach(callback => {
        try {
            callback(data);
        } catch (error) {
            // Avoid a fatal cascading error
            console.error(`[EventBus] Error in listener for '${eventName}':`, error);
        }
    });
}

2. Local Vaults and Promises (KromaDB)

Any normal person would have used localStorage for the data the system saves, but I'm not normal, I'm an idiot. If anyone has already noticed, I obviously reused some of the base logic from Aerko, though adapted. So of course, I could use an IndexedDB wrapper again to make sure that, even if the client is a schizophrenic with 9000 links, the system doesn't run out of space (because localStorage has a 5mb limit).

So I made the schema verification with "onupgradeneeded" force primary keys (keypath, id, etc.), dividing the schemas into 5 stores: "user", "groups", "Categories", "links" and "notepads". Fun fact: if you ask an AI about the data architecture, it'll propose only 2 stores — one for links and one for the notepad. It's possible and it wouldn't even be bad with an array, but it would be just as insane with a single bug.

JAVASCRIPT
// JS/core/db.js
async put(storeName, item) {
    await this.ready; // Hold raw executions to guarantee I/O
    return new Promise((resolve, reject) => {
        const transaction = this.db.transaction([storeName], 'readwrite');
        const store = transaction.objectStore(storeName);
        const request = store.put(item);

        request.onsuccess = () => resolve();
        request.onerror = (e) => reject(e.target.error);
    });
}

3. The Brain: CLI and Regex

This is where the project stands out, because think about it: how many start pages exist? Just on GitHub there are almost 5000 projects, so making another one just for design would be a waste of time. That's why I gave it a little Linux touch and made the search bar a customized CLI (Command Line Interface) optimized for what I do. Besides working as a normal search bar, it also lets you run commands like:

  • "/yt [search]" -> Searches YouTube. Saves me from having to go to YouTube and use the search bar, I just search directly here and it takes me to the results.
  • "/hex [hex color]" -> Takes you to encycolorpedia.es, the color bible.
  • "/svg [search]" -> Searches svgrepo.com. I noticed the other day they use Vercel. I kind of understand it, but it could have been done with Cloudflare, edge computing, wasm, and clients.
  • "/roll [max number]" -> Rolls a die. That's it, as silly as that.
  • "/choose [options]" -> Picks an option from several. For example: "/choose Betis, barça, vardrid" and it automatically picks Betis because the code has good taste, but if you put other things it picks a random one.
  • "/export" -> Exports data
  • "/import" -> Imports data

Besides commands, it also natively supports math operations. I could have used a library, but I preferred to harden the logic with regular expressions, among other things because the most complex calculation I do is "(1440-120)/3". That said, I recommend Mathway for more complex stuff — that page helped me in high school.

JAVASCRIPT
// JS/logic/searchEngine.js
// Parsing and securing raw input
const isMath = /^[0-9+\-*/().\s]+$/.test(query) && /[0-9]/.test(query);
const isOperator = /[+\-*/]/.test(query);

if (isMath && isOperator) {
    try {
        const result = new Function(`return ${query}`)();
        if (result !== undefined && !isNaN(result) && result !== Infinity) {
            const finalResult = Number.isInteger(result) ? result : parseFloat(result.toFixed(4));
            // Spits the calculation inline with the CLI action
            return { action: 'inline', payload: String(finalResult) };
        }
    } catch (e) {
        // In case of incomplete syntax
    }
}

4. Decorators and Raw SVG

Instead of settling for Base64 like a lazy programmer (notice the irony), I completely separated the MIME structure in decorators.js. When you upload an SVG, I use FileReader.readAsText() to extract the original vector directly and embed it in the DOM. Why? Because I'm an idi... I mean, by injecting the svg as raw text into the html, that svg inherits native universal properties like currentColor from the main theme engine. A great way to not go crazy with the project's 5 styles and the SVGs.
