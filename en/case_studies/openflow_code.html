1. Local-First Architecture and Data Sovereignty (IndexedDB)

If you're coming from reading my other case studies, you already know that frameworks (cough, Angular, cough) and I are natural enemies. OpenFlow_V3 is built in pure, hardcore Vanilla JS. But the real gem here isn't that I don't use React or the like — it's that there's no server.

In a traditional website (like Jira), you move a task, your computer sends a little message to the server (who knows where it's hosted), the server processes it, saves it, and replies "okay, done." All that takes a few milliseconds (or seconds if your internet is Movistar on a rainy Sunday). In OpenFlow, your browser is the server. We use the browser's native database: IndexedDB.

IndexedDB is brutally fast, but its native API makes you want to cry because of its horrible callback system. So, with the philosophy of a craftsman (or masochist), I built my own promise-based wrapper in db.js to create a generic async CRUD and add a migration system (to go from v1 to v2 without losing data in the process by adding epics and injecting dates).

The biggest advantage of the Local-First approach is "Data Sovereignty." Your projects live on your machine and don't leave unless you export them. No corporate tracking, no middlemen. And most importantly: real latency is literally zero. Zero loading spinners. If the internet goes down, you keep working as if it were desktop software.

2. The Optimistic Rendering Engine and Collision Radar

Moving a card on a Kanban board seems like the simplest thing, but making it smooth at 60fps without the browser stuttering has its tricks. This is where what I call "Optimistic UI Rendering" comes in. When you drag a card and drop it, the graphical interface updates instantly in the DOM (in just 0ms), while the actual state change is sent to the database in the background, fully asynchronously. It's silent but lethal.

But of course, the native HTML5 Drag & Drop API is a total headache. Mainly because the browser doesn't have a clear sense of where exactly you want to drop the card in the middle of a column (we want manual ordering). So I programmed its own "Radar."

JAVASCRIPT
// js/ui/render.js (Excerpt of the radar's mathematical magic)
function getDragAfterElement(container, y) {
    const draggableElements = [...container.querySelectorAll('.task-card:not(.dragging)')];
    return draggableElements.reduce((closest, child) => {
        const box = child.getBoundingClientRect();
        const offset = y - box.top - box.height / 2; // Distance to the center of the card
        if (offset < 0 && offset > closest.offset) {
            return { offset: offset, element: child };
        } else {
            return closest;
        }
    }, { offset: Number.NEGATIVE_INFINITY }).element;
}

Basically, it's a mathematical function that calculates where the mouse is relative to the vertical center of the other cards in the column, figuring out exactly which card you want to drop it under. Once the radar does its work, the pro version of the drop logic kicks in (the "Benito Villamarín Edition"): the script grabs all the cards in the list, assigns them a new order value based on their current index, and proceeds with saving, so the ranking remains immutable on refresh.

3. Strict Event Delegation (The Security Guard)

If a Single Page Application (SPA) starts slowing down after half an hour of work, it most likely has "Memory Leaks." If you render 200 cards and attach an addEventListener to every little button, you're wasting RAM and creating computational garbage. And if you remove them from the DOM without deallocating the events, goodnight.

To maintain a flat memory profile for people who have OpenFlow open for 8 hours straight, I've applied a strict policy of global Event Delegation.

Looking at the modal logic (modal.js), instead of attaching an event to every sad button to delete epics, the modal-body takes the role of a club bouncer, or rather, of Germán Pezzella cutting off a counterattack. Every click bounces up to it and, through conditional e.target.matches() checks, it decides who to take down. One single guardian, and thousands of temporary elements that are born and die cleanly.

And of course, an unforgivable anecdote in render.js. To prevent Cross-Site Scripting (XSS) attacks when injecting plain text as HTML, I had to wedge in an urgent sanitization function that converts tags like <script> into harmless &lt; entities. "I'm sorry folks, if you don't escape your HTML, you're not Betis material." What can I say, pragmatism in the face of imminent disaster.

4. Taming the Timeline (Frappe) and the Search "Radar"

For the timeline I used the frappe-gantt library. Not about to reinvent the wheel by drawing boxes on CANVAS, so I brought in Frappe with one problem: the usual one — it expects data in one format, and my DB has it in a very different one.

Here, so nobody comes at me, the almighty Adapter Pattern steps in (gantt.js). I map and transform all my OpenFlow tasks on the fly into the structure Frappe expects, with the added bonus of first intercepting the phase (Epic) to steal its hex color and slip it into each timeline bar. And to force my "industrial brutalist palette" into some stubborn SVGs... I use temporary brute-force attacks (or in other words, jamming the styles in via JS after a little setTimeout).

And, to top off the technological crown, there's my real-time search. How does another typical SPA search for things? It sends the string (text) to the backend, does a pseudo-indexed search in the DB, returns results via a REST API, does a massive recalculation, and updates the React DOM. In OpenFlow? You simply have a listener catching your keystrokes, intercepting the DOM nodes one by one and comparing them. Anything that doesn't match... display: none. No network, no async waits, temporary hidden visibility, instant, and computationally free for the interface.

It's rough, free, and reactionary code in every sense, but I can proudly state that the resulting usability justifies every little madness. I hope your head hasn't exploded reading my way of understanding programming. See you in the next interface!
