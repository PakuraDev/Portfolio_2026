Quick Explanation

Alright, imagine building a website is like setting up an exhibition in a museum (like a museum, and the website happens to be a museum — mind-blowing creativity, I know). When building the site, you can take two paths: either you play it safe and decorate with mass-produced furniture and accessories (a framework), or you make everything yourself to your liking (Vanilla JS). On the other hand, you could paint all the pictures directly on the museum wall (a monolithic file), which would be chaos to reorganize, or you could paint each picture on its own canvas and then hang them in the rooms (Web Components).

Well, that's how this website is built. Everything is built with HTML, pure CSS, and Vanilla JS, modularized so that each component (the header, a section, a button, etc.) is an independent file that gets imported and rendered on the site. That said, this isn't optimal, but since it's a single example page to showcase the design, there are no major implications.

Vanilla Architecture and Web Components

Most websites today use frameworks like React, Vue, or Astro (the latter only if you're a gigachad), and I mean, it's comfortable, fast, with everything already built. But my excessive performance philosophy (which borders on madness) prevents me from using them, plus for this website (which, as I said, is just one page with 5 components) it seems like over-engineering (ironic that I say this when I could have put all the components in a single file).

Now, just because I'm using Vanilla JS doesn't mean I'm going to put everything in one file. If you've read any of my repository readmes, you'll know I tend to modularize absolutely everything. That's why each section of the site is made with Web Components and simply imported into index.html with a JS class that extends HTMLElement and renders with this.innerHTML (I know I said I wouldn't get too technical, but honestly, there's no other way to explain this) along with tags like "".

JAVASCRIPT
class="code-keyword">export class AgendaSection extends HTMLElement {
  constructor() { super(); }

  connectedCallback() { this.render(); }

  render() {
    this.innerHTML = `
      <style>
        /* Encapsulated section CSS */
      </style>
      <section class="agenda-layout">
        </section>
    `;
  }
}
customElements.define('agenda-section', AgendaSection);

CSS & Design Tokens

Although the color system looks purely like design, the technical implementation is, as the name suggests, technical (I hope someone found that joke funny). Although I think it's simple, apparently there are people and websites that still don't use variables, so I'm going to brag that my project uses CSS Variables, configured as design tokens, in an independent file.

This allows the website to "breathe" (what can I say, I'm feeling poetic today) under a logical system: from the four shades of "--arena" that mimic the seashore to the institutional "--azul-marino". It's a style architecture that avoids code repetition and allows instant global changes, maintaining coherence across all components.

"Liquid" Buttons

Although they're disabled due to the shadow DOM and architecture used, the buttons have a programmed "Liquid" effect on hover. This, without external libraries, is done with a little technical trick that combines CSS and SVG filters: Inside index.html I defined a hidden global filter that uses "feTurbulence" to generate animated fractal noise. Then, from the component's CSS, I "call" that filter via its ID when the user hovers over the button, although as I said, the Shadow DOM and Web Components don't allow the effect to show. Regardless, so I don't come off as a liar, here's a snippet of the code:

JAVASCRIPT
/* Filter defined in the global HTML */
<filter id="subtle-liquid">
  <feTurbulence type="fractalNoise" baseFrequency="0.03 0.1" numOctaves="1">
    <animate attributeName="seed" class="code-keyword">from="1" to="100" dur="12s" repeatCount="indefinite" />
  </feTurbulence>
</filter>

/* Technical application in the component */
.btn-sorolla:hover .btn-sorolla-fluid {
  opacity: 1;
  filter: url(#subtle-liquid);
}

Developer Pragmatism (or Hack Job)

I'm going to let everyone look at me disapprovingly here, but I also want you to understand. The website is not responsive — access it from a mobile or tablet and it's broken. Could I have made it responsive? Yes, absolutely, but you have to consider two things:

1. Nobody is paying me to make the website: You can all insult me for this one, honestly, it's the most corporate excuse there is, but understand me — it's not a client project, it's an unofficial redesign that people can skip.

2. If I get a cease and desist tomorrow, I'm forced to take it down. We could argue that the odds are very low, and I won't deny that, but as long as they're not zero, I prefer not to dedicate so much time to a vulnerable project.

So yes, you can insult me, call me lazy, and say I'm a hack — I won't deny it because in this case it's technically true. If someday a client asks me to build it (which would mean you're reading this), I'll put in a few extra days and make it responsive. Until then, you'll have to forgive me.
